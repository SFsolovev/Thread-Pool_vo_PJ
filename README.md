# Custom Thread Pool

Это простой учебный пул потоков. Сделан специально без сложной архитектуры, примерно как мог бы сделать джуниор.

## Что реализовано

- `corePoolSize` — сколько потоков создается сразу и сколько стараемся держать минимум.
- `maxPoolSize` — верхняя граница потоков.
- `keepAliveTime` — сколько лишний поток ждет задачу, потом завершается.
- `queueSize` — размер очереди у каждого воркера.
- `minSpareThreads` — если свободных очередей мало, пул заранее создает еще поток.
- У каждого воркера своя `BlockingQueue`.
- Балансировка сделана через Round Robin.
- Есть `execute`, `submit`, `shutdown`, `shutdownNow`.
- Есть простая reject policy.

## Как запустить

```bash
mvn compile exec:java -Dexec.mainClass=ru.example.pool.Main
```

Или без Maven:

```bash
javac -d out src/main/java/ru/example/pool/*.java
java -cp out ru.example.pool.Main
```

## Как работает распределение задач

В пуле есть список воркеров. У каждого воркера своя очередь. Когда приходит новая задача, пул выбирает следующего воркера по Round Robin. Если у выбранного воркера очередь не переполнена, задача кладется туда.

Если свободных воркеров стало меньше, чем `minSpareThreads`, пул пытается создать новый поток, но не больше `maxPoolSize`.

Это не самый умный вариант. Например, Round Robin не всегда равномерно распределяет реальные тяжелые задачи, потому что одна задача может выполняться 100 мс, а другая 10 секунд. Но зато код простой и понятный.

## Политика отказа

Используется `AbortRejectedTaskHandler`. Если все очереди заняты и уже создано `maxPoolSize` потоков, новая задача отклоняется. В лог пишется сообщение, потом выбрасывается `RejectedExecutionException`.

Почему так: это самый простой и честный способ показать вызывающему коду, что сервер перегружен. Задача не теряется молча.

Минусы:

- вызывающий код обязан обработать исключение;
- пользователь может получить ошибку;
- нет backpressure, повторов или graceful degradation;
- для настоящего production-сервера лучше добавить метрики, лимиты, retry или выполнение в вызывающем потоке.

## Логирование

Логируются такие события:

- создание потока;
- добавление задачи в очередь;
- выполнение задачи;
- отказ задачи;
- idle timeout;
- завершение воркера;
- shutdown.

Пример логов:

```text
[ThreadFactory] Creating new thread: MyPool-worker-1
[Pool] Task accepted into queue #1: task-1
[Worker] MyPool-worker-1 executes task-1
[Worker] MyPool-worker-3 idle timeout, stopping.
[Rejected] Task task-30 was rejected due to overload!
```

## Анализ производительности

Стандартный `ThreadPoolExecutor` почти наверняка быстрее и надежнее. Он давно оптимизирован, хорошо протестирован, умеет разные очереди, политики отказа, lifecycle, interruption и мониторинг.

Мой пул проще, поэтому у него есть проблемы:

- синхронизация через `synchronized (workers)` может тормозить при большом числе задач;
- отдельная очередь на поток иногда хуже общей очереди, потому что задачи могут скопиться у одного воркера;
- нет work stealing;
- `shutdown` реализован грубо;
- `Future` самодельный и очень простой.

Плюсы моего варианта:

- легко понять, куда попала задача;
- можно логировать каждую очередь отдельно;
- можно отдельно настраивать балансировку.

## Мини-исследование параметров

Проверялось на демонстрационных задачах со `sleep`. Это не настоящий benchmark, а примерная оценка.

Вариант 1:

```text
corePoolSize=2, maxPoolSize=4, queueSize=5
```

Работает нормально для небольшой нагрузки. Очереди не сразу переполняются, лишние потоки потом умирают по idle timeout.

Вариант 2:

```text
corePoolSize=1, maxPoolSize=2, queueSize=2
```

Быстро начинаются отказы. Подходит только если нужно жестко ограничить нагрузку.

Вариант 3:

```text
corePoolSize=4, maxPoolSize=8, queueSize=20
```

Меньше отказов, но больше потоков и памяти. Если задачи CPU-bound, много потоков может даже ухудшить результат из-за переключения контекста.

Вывод: для `sleep`/IO-задач можно ставить потоков больше, чем ядер процессора. Для CPU-задач лучше держаться около числа ядер. `queueSize` лучше не делать слишком большим, иначе запросы будут долго ждать и пользователь все равно получит плохое время ответа.

## Сравнение с Tomcat/Jetty

Tomcat и Jetty используют намного более продуманные решения. Там есть зрелая обработка соединений, acceptor/worker threads, метрики, graceful shutdown, защита от перегрузки. Этот учебный пул не замена им, а просто демонстрация основных идей: потоки, очереди, балансировка, отказ и завершение.
